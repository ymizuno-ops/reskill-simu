# Project Coding Standards

<naming_conventions>
言語（TypeScriptやPythonなど）を問わず、プロジェクト全体で以下の命名規則を厳守すること。
- **変数名 / メソッド名（関数名）**: `lowerCamelCase`
- **クラス名 / 型定義（Type/Interface）**: `UpperCamelCase` (PascalCase)
- **定数**: `UPPER_SNAKE_CASE`
- **真偽値（Boolean）**: `is`, `has`, `should`, `can` などのプレフィックスを付け、状態が明確にわかる名前にする。
</naming_conventions>

<coding_rules>
- **型安全の徹底**:
  可能な限り静的型付け（TypeScriptの型定義、Pythonの型ヒント等）を行い、`any` や型推論の放棄は行わないこと。
- **早期リターン（Early Return）**:
  ネストを浅く保つため、ガードディレクティブを用いてエラーや異常系は関数の冒頭で `return` または `throw` すること（`else` 句の使用は最小限にする）。
- **マジックナンバーの排除**:
  意味のある数値や固定文字列は直接コードに書かず、定数として定義してから使用すること。
- **エラーハンドリング**:
  例外を握りつぶさないこと。エラー発生時は適切なログを出力し、ユーザーまたは呼び出し元にわかりやすいエラーメッセージを返すこと。
</coding_rules>

<anti_patterns>
以下の実装はコードレビューで弾かれるため、**絶対に生成しない**こと。
1. **1ファイルの肥大化**: 1ファイルが300行を超えるような実装（意味のある単位でコンポーネントや関数に分割すること）。
2. **本番環境でのデバッグ出力**: `console.log()` や `print()` をそのまま残すこと（専用のロガーを使用するか、開発時のみに限定する）。
3. **不要なコメントアウト**: 古いコードをコメントアウトして残すこと（バージョン管理システム(Git)に任せ、不要なコードは削除する）。
</anti_patterns>
